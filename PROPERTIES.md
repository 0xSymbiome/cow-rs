# Properties Registry

This registry is the canonical public index of invariants and state contracts
for `cow-rs`.

Use it with:

- [Verification Guide](docs/verification-guide.md) for how the evidence is
  interpreted
- [Verification Matrix](docs/verification-matrix.md) for the crate and
  workflow lanes that exercise each surface

Executable coverage stays with the crate or browser surface that owns the
behavior. This registry records what must remain true, who owns it, and where
the current evidence lives.

`Covered` uses these values:

- `Yes`: dedicated executable coverage exists
- `Partial`: deterministic coverage exists, but not through a dedicated
  property or state-machine suite
- `No`: the property is registered, but no executable coverage is attached yet

`Last reviewed` records the most recent date the row was confirmed against the
shipped code. The registry follows a 90-day re-review rhythm that mirrors the
dependency-exception policy in `.github/config/deny.toml`: every row is
re-confirmed at least once per 90-day window, and the date here is refreshed
in the same change that touches the owning surface.

## Methodology

The `Type` column uses five labels. Two of them carry distinct evidence
methodologies that reviewers should read with care:

- `Property` rows are backed by a real `proptest!` macro harness with
  committed `tests/proptest-regressions/` seed files. Coverage comes from
  randomized input generation with shrinking to minimal counterexamples
  on failure, and the committed regression seeds preserve failure
  reproducibility across contributors. The codec crates
  (`cow-sdk-core`, `cow-sdk-contracts`, `cow-sdk-signing`,
  `cow-sdk-app-data`) carry their property rows through this lane.
- `Invariant` rows are backed by a curated deterministic boundary sweep:
  a reproducible 128-case iterator over hand-picked boundary cases plus
  named narrow-search profiles that extend the curated set without
  introducing randomness. Coverage is exhaustive over the curated case
  set and is reproducible by design, but it does not discover unexpected
  inputs or shrink to minimal counterexamples. The orchestration crates
  (`cow-sdk-orderbook`, `cow-sdk-trading`, `cow-sdk-subgraph`) carry
  their invariant rows through this lane because the reviewed
  invariants are transport and builder shape rather than roundtrip
  properties.

The remaining three labels — `Contract`, `State machine`, and `Public
API` — cover typed API and parser contracts exercised through targeted
regression tests, browser-wallet session and chain state-machine
transitions covered through deterministic transports and committed
browser automation, and the curated `cow-sdk` facade exported-symbol
snapshot respectively.
